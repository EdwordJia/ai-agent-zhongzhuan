# OpenClaw 开发规则

## 1. 业务冒烟测试规则

### 1.1 触发时机

任何功能开发完成后，在标记任务为完成或提交人类审核前，必须执行业务冒烟测试。

### 1.2 测试范围

至少覆盖以下维度：

1. **功能主路径**：本次改动涉及的核心用户场景是否能正常跑通。
2. **关键依赖**：改动是否影响了上游/下游模块（例如聊天框改动需检查 Agent、MCP、设置页面）。
3. **错误兜底**：无网络、无配置、参数异常等场景是否有明确提示，不崩溃。
4. **回归检查**：确保没有破坏已有功能（优先跑通已有 E2E / 单元测试）。

### 1.3 执行标准

- 冒烟测试不能只看构建是否通过，必须**实际运行应用**并验证业务行为。
- 优先使用自动化脚本（Playwright / 项目自带测试）复现用户操作；无法自动化的场景需手动验证并记录。
- 测试过程中产生的关键截图、日志、网络请求记录应作为交付物保留。

### 1.4 通过标准

- 所有预先定义的冒烟检查项结果为 **通过**。
- 已有自动化测试（`pnpm run build`、`pnpm run test:e2e` 等）全部通过。
- 没有未处理的报错、未捕获的 Promise 异常、控制台 Error。

### 1.5 失败处理

- 冒烟测试不通过，不得提交人类审核或合并代码。
- 必须记录失败现象、复现步骤、初步根因，并重新进入修复流程。

### 1.6 交付物

每次冒烟测试完成后，应在对应任务下补充：

- 测试环境（分支、commit、运行方式）。
- 检查项清单及结果。
- 关键截图或日志路径。
- 是否阻塞上线的结论。

---

## 2. 代码提交前检查清单

- [ ] `pnpm run build` 通过，无 TypeScript 错误。
- [ ] `pnpm run test:e2e` 通过（如项目已接入）。
- [ ] 新增/修改的功能已执行业务冒烟测试。
- [ ] 未引入新的控制台 Error 或构建警告（除非已记录并允许）。
- [ ] 敏感信息（API Key、密码）未硬编码提交（开发期临时写死的需在上线前移除）。

---

## 3. 责任归属

- 功能开发者是冒烟测试的第一责任人。
- 主 Agent 负责监督测试执行并审查交付物。
- 人类审核者拥有最终放行权。
